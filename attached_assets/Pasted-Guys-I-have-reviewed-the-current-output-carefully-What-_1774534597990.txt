Guys,

I have reviewed the current output carefully.

What we have now is a good analysis layer. It detects delays, shows IDs, shows overdue days, groups problems by area, and identifies mismatches between reported progress and expected progress. That part is useful.

But this is still not enough to fully solve Emil’s request.

Right now the system behaves like a reporting and detection tool.

What Emil wants is a practical decision support layer.

That means the next phase must move from:

“What is wrong?”

to:

“What does it mean, what will it affect, and what should I do now?”

This is the real missing layer.

 1.⁠ ⁠The current output is descriptive, not operational

Today the system can say:

this activity is delayed
this activity has 0 percent progress
this activity is overdue by X days
this discipline has N delayed activities

That is good, but Emil already understands the raw schedule.
He does not just need delayed rows listed.
He needs the system to interpret them like an experienced construction planner would.

The next phase must therefore answer 4 things for every serious issue:

Why is this a problem?
What does it block or affect?
How urgent is it compared to other issues?
What should the project manager do about it right now?
 2.⁠ ⁠We need a true root cause layer

Not every delayed activity matters equally.

Some are symptoms.
Some are root causes.

The next phase must distinguish between:

root cause tasks
downstream tasks
secondary affected tasks
noise / lower-priority items

Example:
If EL coordination tasks have not started, and multiple later installation activities depend on them, then the coordination tasks are the real root cause.
The later tasks are consequences.

The system must explicitly say this.

We need the output to include language like:

“Primary root cause”
“Downstream consequence”
“Blocked by unresolved coordination”
“Dependent activities likely to slip if this is not resolved”

Without this, the user still has to interpret everything manually.

 3.⁠ ⁠We need dependency and propagation logic

This is probably the most important missing capability.

Today we show delayed tasks.
Tomorrow we must show delay propagation.

For each critical delayed activity, the system should estimate:

what later tasks are likely affected
which discipline or area is exposed
whether this is an isolated delay or a cascading issue

Even a first simple version is enough.

For example:

“This task appears to block downstream component placement”
“This task delays procurement readiness”
“This issue may affect EL, VVS and later installation sequencing”
“This issue is likely upstream of multiple tasks in the same area”

It does not have to be perfect in version 1.
But it must clearly show that the system understands consequence, not just status.

 4.⁠ ⁠We need prioritization, not just lists

A project manager does not need 34 issues without ranking.

He needs:

what to focus on first
what is mission critical
what can wait
what creates the biggest downstream damage

So the next phase must prioritize issues using clear categories such as:

Critical now
Important next
Monitor only

Or:

Immediate action required
High risk if unresolved within 7 days
Low operational significance

This prioritization should not only be based on overdue days.
It should also consider:

whether the task has 0 percent progress
whether it is an upstream or coordination task
whether it has many likely dependent tasks
whether it belongs to a discipline cluster with multiple simultaneous delays

In other words:
We need a risk-weighted priority score, not just a sorted overdue list.

 5.⁠ ⁠We need action recommendations that are direct and practical

This is where Emil will really feel the difference.

The current assessment is still too generic.
It sounds intelligent, but not yet operational enough.

The next phase must produce recommendations in plain project language.

Not:

“Immediate action should focus on closing these decisions”

But:

“Resolve EL coordination and component placement decisions before allowing downstream installation to continue”
“Do not continue dependent installation tasks until DALI and dimming coordination is clarified”
“Prioritize closure of Sikringsprojekt design baseline because multiple later placements depend on it”
“Escalate unresolved bygherre/coordination decisions this week”
“Review whether downstream work is proceeding on assumptions rather than approved inputs”

Recommendations should feel like instructions from an experienced planner, not generic AI summary text.

 6.⁠ ⁠We need recommended sequence of action

Emil does not only want recommendations.
He wants to know in what order to act.

So for the top issues, the system should produce something like:

Recommended order of action
Resolve coordination / design clarification tasks
Confirm pending bygherre decisions
Freeze downstream tasks that depend on unresolved inputs
Reassess affected downstream schedule clusters
Then release installation work

This is extremely valuable because it turns the output into a plan of action.

 7.⁠ ⁠We need first-generation resource logic

Emil explicitly asked for something more advanced than Copilot, including practical resource thinking.

We do not need full manpower simulation yet.
But we do need the first step.

For major delays, the system should begin to say things like:

“This issue is unlikely to be solved by adding labour only, because it appears to be a coordination / decision dependency”
“This delay may require management attention, not site manpower”
“Additional staffing may help only after prerequisite design inputs are resolved”
“This looks like a planning/coordination bottleneck rather than a production capacity problem”

This distinction is very important.
Because in construction, many delays are not solved by simply adding people.

The system must start showing that it understands that difference.

 8.⁠ ⁠We need to separate task types more intelligently

The next phase should classify delayed tasks into categories such as:

coordination tasks
design clarification tasks
bygherre decision tasks
production tasks
procurement-related tasks
milestone / overview tasks

Because the recommendation depends on the type of task.

Example:
If a task is a design clarification task, the recommendation should not be “add workers”.
If a task is a production task with approved inputs, extra manpower may be relevant.
If a task is a bygherre decision task, escalation is the likely action.

This is how the output becomes believable.

 9.⁠ ⁠We need a “manager view” summary at the top

Before the detailed list, the output should say something like:

Management conclusion

The schedule risk is currently driven primarily by unresolved early-stage coordination and design clarification items, especially in EL/coordination and Sikring.
These are not isolated delays.
They appear to create downstream risk across multiple areas.
The immediate focus should be on closing prerequisite decisions and coordination inputs before pushing later installation activities.

That kind of conclusion makes the system feel senior and valuable.

10.⁠ ⁠What success looks like

We know this phase is successful when Emil can open the analysis and immediately understand:

what the main problem is
what is causing it
what it affects
what he should do first
what not to waste time on
whether this is a manpower issue, coordination issue, or decision issue

If he still has to interpret the data himself, we have not finished the job.

11.⁠ ⁠Practical output structure for next version

The output for each major issue should ideally include:

ID
Activity name
Why flagged
Root cause or consequence
Discipline / area
Severity / priority
Likely downstream impact
Recommended action
Resource implication or decision implication
Recommended time horizon for action

Example:

ID 21
Activity: E100.02 Indretning af produktionskøkken
Status: 163 days overdue, 0 percent progress
Problem type: Upstream design / coordination blockage
Why it matters: blocks equipment data and final layouts required for coordinated MEP work
Downstream impact: likely affects coordinated installation readiness in later stages
Priority: Critical now
Recommended action: close input and design decisions before releasing dependent downstream tasks
Resource note: additional site labour will not solve this until prerequisite design inputs are finalized

This is the kind of output that will make Emil feel we have understood the real need.

12.⁠ ⁠Final direction

So to be completely clear:

The next phase is not about detecting more delayed rows.

The next phase is about transforming schedule data into:

root cause understanding
consequence mapping
priority ranking
practical action guidance

That is the difference between a smart report and a tool a serious project manager will actually use.

This is the layer we need now.
