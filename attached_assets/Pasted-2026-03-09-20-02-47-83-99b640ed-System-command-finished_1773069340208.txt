RAG Agent SaaS
Overview
A Python-based RAG (Retrieval-Augmented Generation) Agent SaaS application that replicates the n8n Azure AI RAG workflow using Supabase pgvector instead of Azure AI Search.

Architecture
Framework: FastAPI
Vector Store: Supabase with pgvector extension
Embeddings: Azure OpenAI text-embedding-3-small
LLM: Azure OpenAI GPT model
PDF Processing: Azure Document Intelligence OCR + LangChain text splitters
Project Structure
src/
├── __init__.py        # Package init
├── config.py          # Configuration and settings
├── database.py        # Supabase/PostgreSQL database operations with SQL injection protection
├── embeddings.py      # Azure OpenAI embedding generation
├── azure_ocr.py       # Azure Document Intelligence OCR for PDF table extraction
├── pdf_processor.py   # PDF processing with Azure Document Intelligence OCR
├── vector_store.py    # Vector store management
├── agent.py           # RAG agent with dual vector store querying
└── main.py            # FastAPI application
API Endpoints
POST /upload - Upload two PDF schedules
curl -X POST "https://your-domain/upload" \
  -F "session_id=session_abc123" \
  -F "old_session_id=table_old_xyz789" \
  -F "old_schedule=@old_schedule.pdf" \
  -F "new_session_id=table_new_xyz123" \
  -F "new_schedule=@new_schedule.pdf"
Field	Description
session_id	Main chat session ID
old_session_id	Vector store table name for old file
old_schedule	The old PDF file
new_session_id	Vector store table name for new file
new_schedule	The new PDF file
POST /query - Query the AI agent
curl -X POST "https://your-domain/query" \
  -F "query=Compare the two schedules" \
  -F "vs_table=session_abc123" \
  -F "language=da" \
  -F "old_session_id=table_old_xyz789" \
  -F "new_session_id=table_new_xyz123" \
  -F "format=html"
Field	Description
query	User's question
vs_table	Main session ID (for chat history)
language	"da" (Danish) or "en" (English)
old_session_id	Reference to old file's vector store
new_session_id	Reference to new file's vector store
format	"markdown" (default) or "html" (premium styled)
GET /health - Health check
Flow
User uploads PDF files via /upload with a session_id
Each PDF gets its own vector store table (e.g., vs_{session_id}_{filename})
Binary PDF is parsed using Azure Document Intelligence OCR (preserves tables/structure)
Text is chunked and embedded via Azure OpenAI
Embeddings stored in Supabase pgvector
User queries via /query with list of vector store tables
Agent retrieves from both stores and provides comparison analysis
Table Extraction (Construction Schedules)
Uses Azure Document Intelligence structured table output (analyzeResult.tables)
Handles merged cells via rowSpan/columnSpan tracking
Table chunks include: markdown + embedded structured JSON
Row chunks include: page_number, cells_data with coordinates
Format: TABLE {id} (Pages [...])\n{markdown}\n[STRUCTURED: {json}]

Table chunks: Full table as markdown + embedded JSON with cell data
Row chunks: Individual rows with page numbers
Page text chunks: Split at 1000 chars with 200 char overlap
Step D → Embedding Generation

Each chunk sent to Azure OpenAI text-embedding-3-small
Returns 1536-dimension vectors
Batched 16 at a time for efficiency
Step E → Store in Supabase

Embeddings + text content stored in the session-specific table
IVFFlat index created for fast similarity search
3. Result
Both PDFs are now searchable vector stores, ready for comparison queries via /query.

Old PDF: OCR (~4s) → Chunk (~0.1s) → Embed (~1s) → Store (~2s)
New PDF: OCR (~4s) → Chunk (~0.1s) → Embed (~1s) → Store (~2s)

Good question. Let me be honest about what the current approach does and its tradeoffs:

What the fix does:
Layer 1 - Smarter chunking (PDF Processor):

Small tables → kept fully intact (markdown + structured JSON)
Medium tables → drops the raw JSON blob (which is huge) but keeps the readable markdown table
Very large tables → splits the markdown into multiple parts
Layer 2 - Safety truncation (Embeddings):

If a chunk still exceeds ~24K characters, it gets truncated before sending to the API
The honest tradeoff:
For extremely large tables, some content at the end of a truncated chunk could be lost from the embedding. However:

The row-level chunks are still created separately — every single row of every table gets its own small chunk with all cell values. These are never truncated because individual rows are small.
So even if a full table chunk gets trimmed, the individual row data is still fully searchable.
The text content (non-table pages) uses 1000-char chunks, which are well within limits.
In practice: No actual data is lost for search/comparison because the row chunks capture everything. The table chunks are just for giving the AI a "big picture" view of the table structure.

Do you need a larger embedding model?
No. The text-embedding-3-small (8K token limit) is fine because:

Row-level chunks are small (~100-300 tokens each)
Text chunks are 1000 chars (~250 tokens each)
Only the full-table overview chunks were too large, and those are now properly split
If you ever wanted to switch, text-embedding-3-large has the same 8K token limit anyway — the difference is embedding dimension (3072 vs 1536), not input size. The chunking strategy is the real solution, and that's now handled properly.
